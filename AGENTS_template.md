# AGENTS.md (HOI4 Modding 汎用テンプレート)

このファイルは、本リポジトリ（Hearts of Iron IV MODプロジェクト）で作業するすべてのAIエージェント（Claude, Gemini, GitHub Copilot, Opencode, Codex等）向けの共通作業指針です。新しいMODプロジェクトを始める際、あるいは既存のプロジェクトにAIを導入する際に、このファイルを `AGENTS.md` にリネームしてプロジェクトのルートディレクトリに配置してください。

*(注意: `<...>` で囲まれた部分は、ご自身のMODプロジェクトに合わせて適宜書き換えてください)*

## 目的 / Purpose
- 本ファイルは、AIエージェント向けの全体的な作業指針です。
- Hearts of Iron IV (HOI4) バージョン `<対応バージョン>` 向けのMOD開発を前提とします。

## 開始時の必須手順 (Required First Steps)
- ファイルを編集する前に、必ず本ファイルを読むこと。
- `git status --short --branch` を実行し、現在のブランチと作業ツリーの状態を確認すること。
- ユーザーの手動変更を上書きまたは無断で取り消さないこと。
- 編集を行う前に、適切な命名規則に従った作業ブランチを作成または切り替えすること。
- `main` ブランチへ直接プッシュしないこと。

## リポジトリ構成
* `<MODのベースディレクトリまたはリポジトリの主要ディレクトリ>` (例: `my_mod/`)
  * `common/`: ゲームルール、国家方針、法律、国民精神、ディシジョン、キャラクター、各種システムスクリプト
  * `events/`: 各国およびゲームシステム用のイベント定義
  * `history/`: ゲーム開始時の初期データ（各国政体、技術、ステート情報、初期軍配置 OOB）
  * `localisation/`: 翻訳定義ファイル（日本語: `japanese/`, 英語: `english/` 等）
  * `gfx/`: スプライト、国旗、顔写真、アイコン等の画像アセット
  * `interface/`: UI定義（.gui）および .gfx 設定ファイル
  * `tests/` / `debug/`: デバッグ用のスクリプトや検証用定義（任意）

## 開発・プレイテストフロー
- 開発したMODは、HOI4ランチャーでMODを有効化し、ゲーム内でプレイテストを行います。
- 必要に応じて、以下のコンソールデバッグコマンドを活用してリアルタイムに動作検証を行ってください。
  * `tdebug`: デバッグ情報表示（マウスオーバーした州、国家のID、変数、フラグ、アクティブな補正等を表示）
  * `event <event_id> <TAG>`: 指定した国家に対して特定のイベントを強制実行
  * `reload interface`: GUI定義（.gui/.gfx）の変更を即時ゲーム内にロード・反映
  * `reload localisation`: ローカライズ（.yml）の変更を即時反映（※ゲームの動作が一時的に停止する場合があります）
  * `imgui show profiler`: Scriptタブ等から、特に負荷の高い `hourly`（毎時処理）の処理時間をプロファイリング

## ファイル配置の原則
1. **既存構成の維持:** プロジェクト固有のディレクトリ構造やファイル名のルールを崩さないこと。
2. **適切な配置:** 新規ファイルを作成する際は、同系統の既存ファイルが並んでいるディレクトリに配置すること。
3. **ファイルの分割:** 1つのファイルに膨大なロジックを詰め込むのを避け、機能やシステム、国家タグ（TAG）ごとに小さく分割すること。

## コーディング規約 (共通)
- **インデント:** スペース2個（タブは使用禁止）。
- **可読性:** Paradox scriptの可読性を保つため、原則「1行に1命令」とする。
- **コメント:** 必要最低限に留め、内容は日本語で記述してよい。
- **改行・フォーマット:** 既存コードの改行位置や中括弧 `{ }` の記述スタイルを維持する。

## 命名規則
- **国別ファイル:** 3文字の国タグ（TAG）を使用する（例: `GER.txt`, `JAP_focus.txt`）。
- **システム共通ファイル:** 接頭辞としてアンダースコア `_` や特定のシステム名をつける（例: `_economy_spheres.txt`）。
- **GUI関連:** `.gui` ファイル名と、それを制御する `scripted_guis` の定義名を一致させる。
- **変数・フラグ:** 用途が分かりやすい英語のスネークケースで統一する。

## ローカライズ規約
- **保存場所:** `localisation/japanese/`（日本語）や `localisation/english/`（英語）等の該当言語フォルダ。
- **形式:** YAMLフォーマット。ファイルの先頭行に `l_japanese:` や `l_english:` などのヘッダーを必須とする。
- **エンコーディング:** **必ず「UTF-8 with BOM (BOM付き UTF-8)」**で保存すること。BOMがない場合、ゲーム内で文字化けや読み込みエラーが発生します。
- **変数表示:** テキスト中に変数を埋め込む際は `[?variable_name]` の記述形式を使用する。

## スクリプト最適化・パフォーマンス指針
- **早期リターン (Early Return):** 条件ブロック（`trigger`, `limit` 等）では、負荷の低い単純な条件判定（`has_country_flag`、`tag = GER` 等）を先頭に書き、重い条件判定（変数の比較、広範なスコープ探索など）は後回しにする。
- **毎時処理 (`hourly`) の制限:** 毎時実行される判定や処理はゲーム動作を著しく重くするため、極力 daily（毎日処理）やフラグ変更時のイベントトリガー方式に変更すること。
- **重いスコープの回避:** 頻繁に実行されるコード内で、`any_state` や `any_country` などのグローバル探索スコープを使用しないこと。
- **キャッシュの活用:** ステートの変更や変数の計算結果は、state_flag や country_flag に記録してキャッシュし、毎 tick 判定が再計算されるのを防ぐ。

## アセットに関する規約
- 音声ファイル（`.wav`, `.ogg`）は、指示がない限り絶対に変更・削除しないこと。
- 画像アセットを追加する際は、適切なサイズ・アルファチャンネル（透過）に最適化し、既存の命名スタイルを崩さないこと。

## GitFlow / ブランチポリシー
MODの共同開発において、ブランチの運用は以下のルールに従ってください。
- `develop` ブランチを統合・開発用ブランチとして使用し、作業ブランチは `develop` にマージする。`main` へ直接プッシュしないこと。
- ブランチ名は以下の形式で命名すること。
  ```text
  type/scope_name
  ```
  * `type` の例: `feature`（新機能・新国家等）、`fix`（バグや記述の修正）
  * `scope_name` の例: `GER`（特定の国に関する作業）、`_map`（マップ関連）、`_system`（共通システム関連）
  *(例: `feature/JAP_rebellion_tree`, `fix/crash_event_trigger`)*
